Momentra
REST API Specification & Screen–SQL Mapping
MVP v1.0  ·  Multi-tenant  ·  PostgreSQL backend
Base URL:  https://api.momentra.app/v1
Auth:  Bearer JWT in Authorization header. Tenant resolved from JWT tenant_id claim.
Content-Type:  application/json  for all request/response bodies.
Pagination:  Cursor-based via ?cursor=&limit= query params. All list endpoints return { data: [], next_cursor, total }.

5. SQL Trigger Functions Called by API
The Momentra SQL layer includes deterministic trigger functions. The API layer does NOT duplicate this logic — it calls or relies on these functions directly.
Function / Trigger
Called When
What It Does
recompute_commitment_rollup()
After any INSERT into contribution_transactions
Recomputes fulfilled_amount on commitments; re-derives commitment status (PENDING/PARTIAL/FULFILLED/OVERDUE)
recompute_moment_health()
After commitment status changes or milestone updates
Inserts new row in moment_health_snapshot with weighted scores; sets health_state (STABLE/BALANCED/DRIFTING/AT_RISK)
evaluate_signals()
After health snapshot or on scheduled job
Upserts/resolves signals: FUNDING_GAP, OVERDUE_COMMITMENT, TIMELINE_DRIFT, INACTIVITY, IMBALANCE, MOMENTUM_SLOWDOWN
log_event()
Called explicitly by API after every state-changing operation
Inserts audit record in event_logs — powers Activity Flow screen and lifecycle history
trg_set_updated_at()
BEFORE UPDATE on moments, commitments, moment_participants
Automatically sets updated_at = now() — no API layer code needed
derive_commitment_status()
Immutable SQL function used inside recompute_commitment_rollup
Returns PENDING | PARTIAL | FULFILLED | OVERDUE from committed_amount, fulfilled_amount, due_date
